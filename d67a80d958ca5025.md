# LinkPilot

LinkPilot 是一个面向技术团队与内容研究者的外链资源归集与导航工具。该项目将分散于各类技术博客、新闻站点、手册文档中的高质量文章链接进行结构化整理，通过统一的条目索引与分类体系，帮助用户快速定位特定主题下的权威资料。LinkPilot 本身不存储或转发文章内容，仅提供链接的元信息管理与检索能力，适用于需要长期维护外部知识库、构建团队技术阅读列表或进行文献回溯的场景。

## 功能概览

- 批量链接归集：支持将大量文章 URL 导入系统，自动提取域名、路径参数等元数据，并按来源站点与内容类型进行初步分类。

- 多级标签体系：允许用户为每条链接添加自定义标签，支持层级标签结构，便于从技术领域、难度等级、适用岗位等多个维度筛选资源。

- 链接状态监控：定期检测已收录链接的可访问状态与响应码，标记失效或重定向的条目，帮助维护资源库的健康度。

- 全文检索与筛选：提供基于标题、域名、标签、收录时间的组合检索能力，支持正则表达式过滤与批量导出搜索结果。

- 导入与导出兼容性：支持从 CSV、JSON、Markdown 列表等常见格式导入链接数据，导出结果可适配为书签文件、阅读清单或技术周报素材。

- 访问统计快照：记录每条链接的点击次数、最近访问时间与引用频次，为内容价值评估提供基础数据支撑。

- 团队协同注解：允许多用户对同一条链接添加注释与评级，注解内容支持版本记录，便于团队内知识传递。

- 定时同步任务：可配置定时任务从指定 RSS 源或站点地图自动发现新文章链接，纳入待审核队列，减少人工维护成本。

## 应用场景

技术团队内部知识库构建：研发负责人可使用 LinkPilot 归集团队成员推荐的架构设计、故障复盘、性能优化类文章，按服务模块打标后分发给对应小组，形成持续更新的团队学习地图。

技术社区内容聚合：开源项目维护者可通过 LinkPilot 收集社区用户撰写的教程、案例与集成方案，将优质外链整合为项目官网的生态资源页，降低新用户的上手门槛。

个人技术阅读管理工作流：开发者可将日常浏览中发现的优质博文、规范文档与工具手册一键录入 LinkPilot，利用标签与检索功能建立个人技术档案库，避免碎片化阅读导致的信息遗失。

技术文档撰写参考资料整理：技术作家在编写系统设计文档或白皮书时，使用 LinkPilot 快速收集并分类相关标准、论文与官方公告链接，确保引用来源可追溯且便于后续校验。

## 快速开始

以下命令演示如何在本地环境中获取 LinkPilot 源码、安装依赖并启动开发服务。

```bash
git clone https://github.com/linkpilot/linkpilot.git
cd linkpilot
npm install
npm run dev
```

执行上述命令后，开发服务器将默认运行于 127.0.0.1:3000。请确保本地 3000 端口未被占用。若需修改端口，可编辑项目根目录下的 .env 文件中的 PORT 字段。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Node.js | 18.x 或 20.x LTS | 运行时环境，建议使用 nvm 管理多版本 |
| npm | 9.x 或 10.x | 依赖管理与脚本执行工具 |
| PostgreSQL | 14.x 或 15.x | 主数据存储，需启用 pgcrypto 扩展 |
| Redis | 7.x | 用于缓存与定时任务队列，可选但推荐 |
| TypeScript | 5.x | 编译时类型检查，全局安装或使用项目本地依赖均可 |
| Docker | 20.x 或更高 | 仅在使用容器化部署方式时需要 |
| Git | 2.40 或更高 | 源码克隆与版本控制 |

## 项目结构

```
linkpilot/
├── apps/
│   ├── web/                         # 主 Web 应用，基于 Next.js App Router
│   │   ├── src/app/                 # 页面路由与布局组件
│   │   └── src/components/          # 可复用的 UI 组件库
│   └── worker/                      # 后台任务进程，负责链接状态检测与定时同步
│       ├── src/queues/              # Bull 队列定义，包含检测、导入、通知等队列
│       └── src/processors/          # 各队列的任务处理器实现
├── packages/
│   ├── core/                        # 核心业务逻辑层，与数据库和缓存交互
│   │   ├── src/entities/            # TypeORM 实体定义，映射 PostgreSQL 表结构
│   │   ├── src/services/            # 链接管理、标签管理、统计等服务类
│   │   └── src/utils/               # URL 规范化、哈希计算、校验工具
│   ├── api-client/                  # 供外部调用的 REST API 客户端封装
│   └── shared-types/                # 前后端共享的 TypeScript 类型与接口定义
├── docker/                          # 容器化相关文件
│   ├── development/                 # 开发环境 Docker Compose 配置
│   └── production/                  # 生产环境 Dockerfile 与健康检查脚本
├── docs/                            # 完整文档源文件，采用 Markdown + Mermaid
│   ├── quick-start.md               # 快速入门指南
│   ├── api/                         # API 接口文档，按模块拆分
│   └── operations/                  # 运维部署与监控手册
├── scripts/                         # 辅助脚本，包含数据迁移、种子填充与备份
├── tests/                           # 单元测试与集成测试，使用 Jest 与 Supertest
├── .env.example                     # 环境变量模板文件
├── docker-compose.yml               # 主服务编排文件
├── package.json                     # 项目依赖与工作区脚本定义
├── tsconfig.base.json               # 基础 TypeScript 编译配置
└── README.md                        # 项目入口说明文档
```

## 贡献指南

1. 查阅问题列表与项目看板：访问 GitHub Issues 页面，筛选标记为 help-wanted 或 good-first-issue 的条目，确认无重复工作后认领任务。在看板中将自己添加为受理人。

2. 派生仓库并创建功能分支：从主仓库派生至个人账户，基于 main 分支创建命名规范为 feature/功能简述 或 fix/问题编号 的新分支，禁止直接向 main 分支提交。

3. 编写代码并确保测试通过：遵循项目根目录下的 .eslintrc 与 .prettierrc 规则进行代码格式化。新增功能需在 tests 目录下补充对应的单元测试或集成测试，确保全部测试套件通过且覆盖率不低于现有水平。

## 常见问题

Q: 系统支持导入的链接数量上限是多少？
A: 系统本身不设硬性上限，实际承载能力取决于部署环境的 PostgreSQL 性能与磁盘空间。在推荐配置（4 核 CPU、16GB 内存、SSD 存储）下，单表可稳定支持 500 万条链接记录。当链接数量超过 100 万时，建议按收录时间进行表分区或启用只读归档节点。

Q: 如何迁移已有的书签或收藏夹数据到 LinkPilot？
A: 项目提供了迁移脚本 tools/migrate-bookmarks.js，支持从 Chrome 导出的 HTML 书签文件、Firefox JSON 备份以及 Pocket CSV 导出格式。执行 npm run migrate -- --source=chrome --file=./bookmarks.html 即可开始导入。迁移过程中会保留原始文件夹结构并自动转换为标签体系。

Q: 链接状态监控是否会影响源站点的访问统计？
A: 状态监控请求在 HTTP 头中设置了 User-Agent 为 LinkPilot-HealthCheck/1.0，并采用 HEAD 方法优先，仅当 HEAD 请求失败时才降级使用 GET 且不下载响应体。每次检测间隔不低于 24 小时，且所有请求均携带 Cache-Control: no-cache 以尊重源站的缓存策略，不会对源站正常流量产生可观测的影响。
