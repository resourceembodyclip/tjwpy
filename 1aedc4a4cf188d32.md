# WebIndex 移动端技术资源导航站

WebIndex 是一个面向移动端开发者和技术研究人员的结构化外链资源聚合系统。该项目定位于解决移动端开发文档碎片化、技术文章分散以及优质资源检索效率低下的问题，通过对特定领域的技术文章、API 参考和案例代码进行系统化收录与分类，为开发者提供可检索、可追溯的技术资源索引服务。

本项目不生产原创内容，而是通过人工筛选与自动化校验相结合的方式，将散落在互联网各处的移动端技术资料按照统一元数据模型进行重整。当前批次为第 20/56 批资源收录批次，共计 180 个资源链接，涵盖移动端 H5 开发、跨端框架、性能优化、兼容性处理及工程化工具等多个技术子领域。

目标用户包括移动端 Web 应用开发工程师、技术团队负责人、技术选型评估人员以及计算机相关专业的研究生与高年级本科生。项目本身不依赖特定前端框架，可独立部署为静态站点或集成到现有文档系统中。

## 功能概览

**资源链接归集与分类**：系统根据 URL 模式、来源域名及内容指纹对收录的链接进行自动归类，支持按技术领域、发布时间和内容类型进行多维度筛选。

**元数据提取与摘要生成**：对每个收录的链接自动抓取页面标题、主要章节标题及代码片段数量，生成 160 字符以内的内容摘要供快速预览。

**链接可用性周期性校验**：内置基于 HTTP 状态码的链接存活检测任务，对失效链接、重定向链接和响应超时链接进行标记并生成巡检报告。

**全文检索与快速定位**：基于倒排索引机制对资源标题、描述及分类标签建立索引，支持布尔查询和模糊匹配，平均检索响应时间低于 300 毫秒。

**批次管理与版本追踪**：每个收录批次包含资源清单、收录时间戳和审核状态，支持按批次维度导出资源列表为 JSON 或 Markdown 格式。

**自定义标签体系**：允许用户为资源添加自定义标签，并基于标签关系构建知识图谱式的资源关联网络。

**暗色模式与移动端适配**：前端展示层针对移动设备触摸交互和屏幕尺寸进行响应式设计，提供明暗两种主题并跟随系统偏好自动切换。

**开放 API 接口**：提供 RESTful 风格的资源查询接口，支持分页、排序和字段过滤，便于第三方工具集成调用。

## 应用场景

移动端项目技术选型时的参考信息检索。当团队需要评估某一移动端框架或组件库的社区活跃度与生产案例时，可通过本系统检索相关的技术文章和案例链接，从实践角度辅助选型决策。

日常开发中遇到特定兼容性问题的快速排查。开发者可通过关键字检索历史技术文章，查找同类问题的解决方案或相关讨论，减少重复性的问题定位时间。

技术团队内部知识库的素材来源。技术负责人可将本系统收录的高质量资源链接导入团队 Wiki 或文档平台，作为内部培训材料或技术周报的内容素材。

学术研究中移动端技术发展脉络的文献梳理。研究生或科研人员可通过批次时间轴和分类标签，追踪移动端特定技术方向在一定时期内的演进路径和关键贡献点。

## 快速开始

以下步骤适用于在本地环境部署 WebIndex 静态站点或运行开发预览服务。

```bash
# 克隆项目仓库至本地
git clone https://github.com/webindex/webindex-starter.git webindex

# 进入项目根目录
cd webindex

# 安装项目依赖（使用 npm 或 yarn）
npm install

# 或使用 yarn
yarn install

# 启动开发服务器，默认监听端口 3000
npm run dev

# 构建生产环境静态文件输出至 dist 目录
npm run build
```

完成上述步骤后，访问本地服务地址即可查看资源导航页面。若需更新资源索引数据，将最新批次的链接清单放置于 `data/batches/` 目录下，执行 `npm run index` 命令触发索引重建流程。

## 贡献指南

贡献者需遵循以下流程参与项目资源收录与代码改进。

第一步，查阅项目 Issue 列表及 Projects 看板，确认当前批次收录进度和待办任务。对于新增资源收录请求，需先提交 Issue 说明资源来源、所属技术领域及收录理由，由项目维护者评估后分配收录编号。

第二步，克隆项目仓库并在本地创建功能分支。分支命名遵循 `feature/resource-batch-{批次号}` 或 `fix/{问题简述}` 格式。对于资源数据更新，需在 `data/batches/` 目录下按模板格式新增 JSON 文件，确保每条资源包含 `url`、`title`、`category` 和 `tags` 字段。

第三步，执行本地索引重建和链接校验。运行 `npm run index` 命令更新索引文件，运行 `npm run validate` 对新收录链接进行可用性检测。所有校验失败的链接需在提交前排除或附加备注说明。

第四步，编写或更新对应的单元测试用例。新增功能或修复缺陷时，需在 `tests/` 目录下补充测试代码，确保测试覆盖率达到 80% 以上。运行 `npm run test` 确认全部测试用例通过。

第五步，提交 Pull Request 并填写变更摘要模板。摘要需包含变更类型、影响范围、测试结果和关联 Issue 编号。项目维护者在 Code Review 通过后合并至主分支，并触发自动化部署流程更新线上索引。

## 常见问题

问：如何请求收录某个技术资源链接？
答：请在本项目 Issue 页面提交收录请求，标题格式为「[收录请求] 资源名称」，正文中提供完整 URL、资源简述和推荐分类标签。项目维护者将在 5 个工作日内评估并回复是否纳入后续批次。

问：收录的链接失效了如何处理？
答：项目内置了定期链接校验任务（每周六 UTC 0 点执行）。若用户发现失效链接，可通过 Issue 或项目讨论区报告，标注「链接失效」并提供 URL。失效链接会被标记为 `unreachable` 状态并从主索引中降权处理，连续三次校验失败后将移至归档目录。

问：能否将本项目部署到内网环境使用？
答：可以。本项目所有静态资源均可离线部署，不依赖外部 CDN。只需将 `dist/` 目录下的所有文件拷贝至内网 Web 服务器，并确保 `config/production.json` 中的 `apiBaseUrl` 指向内网 API 地址即可。索引数据需提前生成并随部署包一并分发。
