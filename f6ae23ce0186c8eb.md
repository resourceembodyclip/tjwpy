# LinkNavigator

LinkNavigator 是一个面向技术调研、内容聚合与知识管理场景的轻量级外链资源汇总平台。该项目定位于帮助开发者、技术文档撰写者、数据分析师以及运维工程师，将散落在多个来源的 URL 资源进行结构化收录、分类展示与快速检索。LinkNavigator 本身不存储文章内容，而是作为一份可自部署的导航索引，将原始链接按项目批次、业务主题或技术领域进行组织，从而解决人工收藏夹层级混乱、跨团队共享困难、链接失效不可追踪等实际问题。通过提供固定的文档模板与自动化构建流程，该项目可作为团队内部知识库的入口层，也可作为个人技术栈的补充索引。

## 功能概览

**按批次归档的外链索引**：每批资源链接以固定数量（如本批次 180 个）聚合为一个发布单元，支持版本化浏览与历史回溯。

**原始链接透传展示**：所有收录的 URL 均以原始字符串形式直接呈现，不进行协议补全、域名规范化或跳转包装，确保链接的真实性与可审计性。

**多维度分类浏览**：资源列表章节按技术领域、应用场景或内容类型划分小节，便于用户按需定位相关链接。

**轻量级本地预览服务**：提供基于 Node.js 或 Python 的静态服务器，可在本地启动后通过浏览器查看完整的资源索引页面。

**依赖环境检测与版本提示**：通过安装要求表格明确列出 Node.js、npm、Git 等必需组件的版本下限，降低部署门槛。

**文档导航表格**：按使用层面（新手入门、配置调优、API 参考、运维监控）划分文档目录，帮助不同角色快速找到对应资料。

**ASCII 目录树可视化**：在项目结构中展示完整的目录层级与文件用途注释，方便贡献者理解代码组织方式。

**标准化贡献流程**：提供 Fork、分支、提交、推送、发起 Pull Request 的五步协作规范，降低外部贡献者的参与成本。

## 应用场景

**技术调研期间的临时链接收藏**：当研发团队需要对某一技术方向（如前端框架、数据库中间件、云原生工具）进行横向对比时，可将候选文档、官网、评测文章的链接统一收录至 LinkNavigator 的某一批次中，形成可共享的调研底稿。

**文档站的外链附录管理**：技术文档站或 API 参考手册通常需要引用外部规范、SDK 下载页或社区讨论帖。LinkNavigator 可作为独立的附录服务，与主文档站通过超链接互相关联，降低主站因外链变更而频繁改版的风险。

**运维故障案例库的索引层**：运维团队在排查线上问题时，常参考内部故障报告、外部社区 issue 或云厂商状态页。通过将相关链接按故障类型（网络、存储、容器、配置）分批录入，可逐步构建起可检索的事故知识索引。

**开源项目 README 的外链补充**：开源项目的主 README 不宜堆砌大量外部链接，以免影响核心说明的可读性。LinkNavigator 可作为附属资源库，在主 README 中仅保留一个指向当前批次资源列表的引用，其余链接全部迁移至本项目中统一维护。

## 快速开始

以下命令序列适用于 Linux、macOS 及 Windows WSL 环境，用于在本地克隆项目、安装依赖并启动预览服务。

```bash
git clone https://github.com/your-org/link-navigator.git
cd link-navigator
npm install
npm run build
npm start
```

执行完成后，打开浏览器访问 http://localhost:3000 即可查看当前批次的资源索引页面。若使用 Python 静态服务器，可替换为以下命令：

```bash
git clone https://github.com/your-org/link-navigator.git
cd link-navigator
python3 -m http.server 8000
```

## 项目结构

```
link-navigator/
├── data/                                   # 数据层，存放所有原始链接与批次元信息
│   ├── batches/                            # 按批次拆分的 JSON 文件
│   │   ├── batch_24.json                   # 第 24 批的完整链接列表（180 条）
│   │   └── batch_manifest.json             # 所有批次的索引（编号、链接数量、导入日期）
│   └── schemas/                            # JSON Schema 定义，用于校验链接格式
│       └── link-record.schema.json         # 规定每条记录必须包含 url、source、tags 字段
├── src/                                    # 源代码目录
│   ├── core/                               # 核心业务逻辑
│   │   ├── parser.js                       # 解析原始 URL 列表，提取域名、路径参数
│   │   ├── validator.js                    # 校验 URL 协议、长度、非法字符
│   │   └── aggregator.js                   # 按标签或批次聚合链接，生成分类树
│   ├── templates/                          # 模板引擎相关
│   │   ├── index.template.html             # 主页 HTML 骨架，包含动态渲染占位符
│   │   └── renderer.js                     # 将 JSON 数据填充至模板，输出完整 HTML
│   ├── server/                             # 本地预览服务
│   │   ├── app.js                          # Express 或 http 模块启动入口
│   │   └── router.js                       # 路由配置：首页、批次详情页、健康检查
│   └── utils/                              # 通用工具函数
│       ├── file-helper.js                  # 异步读写文件、目录创建
│       └── logger.js                       # 带时间戳的控制台日志输出
├── docs/                                   # 项目文档，面向使用者和贡献者
│   ├── quick-start.md                      # 快速开始指南（含常见问题排查）
│   ├── configuration.md                    # 环境变量与构建参数说明
```

2.  新建一个功能分支，分支名称需体现改动意图，例如 `feat/add-import-csv` 或 `fix/validate-url-case`。禁止在主分支上直接提交。

3.  在 `data/batches/` 目录下添加或修改批次 JSON 文件时，请严格遵守 `schemas/link-record.schema.json` 中定义的字段类型与必填项。新增链接需包含 `url`、`source`（固定为 "bibjqr"）以及至少一个 `tags` 标签。

4.  提交代码前，请运行 `npm run test` 确保所有单元测试与集成测试通过，并执行 `npm run lint` 检查代码风格。提交信息应遵循 Conventional Commits 规范，使用 `feat:`、`fix:`、`docs:`、`chore:` 等前缀。

5.  将本地分支推送至远程个人仓库，然后通过 GitHub 界面发起 Pull Request 到主仓库的 `main` 分支。PR 描述中需明确说明本次改动的目的、影响范围以及是否涉及破坏性变更。等待至少一位维护者审核通过后即可合并。

## 常见问题

**问：项目是否必须联网才能使用？**

答：LinkNavigator 本身是一个静态索引生成工具，所有 URL 数据以 JSON 格式存储在本地。构建与服务预览过程完全离线，无需访问外部网络。但用户在实际点击资源列表中的链接时，需要设备具备访问目标域名（即 h5.mobile.bibjqr.cn）的网络能力。

**问：如果目标链接失效或域名变更，项目如何处理？**

答：LinkNavigator 只负责收录和展示原始 URL，不主动发起请求检测链接可用性。建议维护者定期（如每季度）通过脚本批量检查所有收录链接的 HTTP 状态码，并在发现失效链接时在批次 JSON 中标记 `status: "dead"` 字段。项目提供了 `validator.js` 中的 `validateUrlStatus` 函数作为辅助检测工具。

**问：能否在同一项目中管理多个不同来源的链接？**

答：可以。`batch_manifest.json` 中的每条记录支持 `source` 字段，用于区分链接来源。当前版本预设了 "bibjqr" 作为来源标识，但贡献者可自由扩展，例如 "github"、"stackoverflow" 或 "internal-wiki"。模板渲染器会根据 `source` 自动添加分组标题。
