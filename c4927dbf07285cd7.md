# WebLink Navigator

WebLink Navigator 是一个面向技术研究者、内容聚合者与信息管理工程师的轻量级外链资源汇总与导航系统。该项目定位于对分散于多源、多批次的文章链接进行统一收录、分类展示与快速检索，解决技术文档编写者、开源项目维护者以及知识库管理员在批量处理外部引用链接时面临的链接分散、难以追踪、缺乏结构化描述等问题。

本项目不提供爬虫或自动化采集功能，而是作为人工筛选与整理后的高质量外链索引平台，强调链接的可读性、归类清晰性与访问稳定性。通过静态化的目录结构与轻量级前端交互，使用者可以快速定位到特定批次、特定领域或特定编号范围内的文章资源，适用于搭建个人或团队内部的技术参考手册、项目文档附录或行业动态聚合页。

## 功能概览

**多批次资源索引**：支持按项目批次（如第 13/56 批）对数百个外部链接进行分组管理，便于跟踪采集进度与覆盖范围。

**原始链接直出展示**：所有收录的 URL 均保持用户提供的原始格式（含协议类型、大小写、扩展名），不进行任何自动补全或规范化改写，确保与源站精确对应。

**分类导航视图**：根据链接来源域名、路径结构或内容主题自动归类，提供按文章编号范围、资源类型（如技术文档、行业报告、案例研究）的快速筛选入口。

**结构化的文档树**：采用 ASCII 目录树清晰展示项目文件布局，使维护者无需阅读代码即可理解资源组织方式。

**依赖与环境透明化**：通过表格形式明确列出运行环境、核心依赖及必需配置，降低部署门槛。

**贡献流程标准化**：提供新增链接、更新分类、修复失效地址的具体操作指南，支持多人协作维护资源库。

**常见问题自助排查**：针对链接访问失败、批次号命名规则、资源收录标准等高频疑问提供预设解答。

**MIT 开源许可**：允许自由使用、修改、分发，仅需保留版权声明，适合嵌入各类技术文档项目。

## 应用场景

技术文档附录管理
开源项目或企业内部技术文档常包含大量外部参考链接。WebLink Navigator 可作为独立子模块嵌入文档仓库，将散见于各章节的引用链接统一迁移至导航系统，避免主文档过于冗长，同时提供链接状态的集中检视入口。

行业资讯每日聚合
资讯编辑或社区运营人员可将每日收集的行业动态、产品发布、技术博客等链接按批次录入系统，生成带编号的资源清单，便于后续周报、月刊的素材整理与回溯。

知识库外链备份与审计
知识库管理员可定期将第三方参考资料导入导航系统，结合批次号与文章编号形成审计轨迹。当外部链接发生变更或失效时，可快速定位影响范围并执行替换或归档操作。

离线文档资源索引
在无法直接访问外网的内网开发环境中，团队可将已下载的离线文档或内部镜像链接通过本系统建立索引目录，方便成员按编号查找对应资源，减少重复沟通成本。

## 快速开始

以下步骤帮助您在本地环境中快速启动 WebLink Navigator 实例。

```bash
# 克隆项目仓库至本地
git clone https://github.com/weblink-navigator/weblink-navigator.git

# 进入项目根目录
cd weblink-navigator

# 安装依赖（基于 Node.js 18+ 与 npm）
npm install

# 启动开发服务器，默认监听 http://localhost:3000
npm run dev
```

执行上述命令后，访问本地服务地址即可浏览当前批次（第 13/56 批）的所有资源链接。生产环境部署请参考 `docs/deployment.md` 中的 Nginx 或 Docker 配置示例。

## 项目结构

项目采用模块化静态生成架构，核心代码与资源数据分离，便于维护和扩展。

```
weblink-navigator/
├── src/                                 # 源代码主目录
│   ├── assets/                          # 静态资源文件（图片、字体、全局样式）
│   │   ├── styles/                      # 全局 CSS 与主题变量定义
│   │   └── images/                      # 界面图标与占位图
│   ├── components/                      # 可复用的 UI 组件库
│   │   ├── LinkTable/                   # 链接列表与分页表格组件
│   │   ├── CategoryFilter/              # 分类筛选与标签组件
│   │   └── BatchNavigator/              # 批次切换与进度指示组件
│   ├── data/                            # 资源数据存储与处理逻辑
│   │   ├── batches/                     # 按批次号拆分的 JSON 数据文件
│   │   ├── index.js                     # 数据索引与导出模块
│   │   └── validator.js                 # 链接格式校验与去重工具
│   ├── pages/                           # 路由页面入口（基于文件系统路由）
│   │   ├── index.jsx                    # 首页：批次总览与快速入口
│   │   ├── batch/                       # 单批次详情页动态路由
│   │   └── search/                      # 全文检索结果页
│   ├── hooks/                           # 自定义 React Hooks（状态管理、本地存储）
│   ├── utils/                           # 通用工具函数（日期格式化、URL 处理）
│   └── config/                          # 站点配置（站点名称、默认批次、分页大小）
├── public/                              # 构建输出目录（由构建脚本自动生成）
├── docs/                                # 项目文档（用户手册、维护指南、架构说明）
├── scripts/                             # 自动化脚本（数据导入、链接健康检查）
├── tests/                               # 单元测试与集成测试用例
├── .gitignore                           # Git 忽略文件清单
├── package.json                         # 项目依赖与脚本命令定义
├── README.md                            # 项目说明文档（本文件）
└── LICENSE                              # MIT 许可证文本
```

## 贡献指南

我们欢迎并鼓励社区贡献，包括但不限于新增链接收录、修正分类错误、优化界面交互、补充文档内容。请按照以下流程提交贡献。

第一，在 GitHub 上 Fork 本仓库至个人账户，并克隆到本地开发环境。确保本地 Node.js 版本符合安装要求。

第二，新建一个功能分支，分支命名规范为 `feature/简述变更内容` 或 `fix/简述修复问题`。例如 `feature/add-batch-14-links`。

第三，在 `src/data/batches/` 目录下按照已有批次文件的 JSON 结构新增或修改链接数据。对于新增链接，请确保每条记录包含 `id`（文章编号）、`url`（原始链接）、`category`（分类标签）和 `title`（可选标题）。提交前运行 `npm run validate` 校验数据格式。

第四，提交变更并推送至个人远程仓库，然后向主仓库的 `main` 分支发起 Pull Request。PR 描述中需明确说明变更目的、涉及的数据范围以及是否经过本地构建测试。

第五，项目维护者将在 3 个工作日内审查 PR，可能提出修改意见或要求补充链接的可访问性截图。通过审查后，变更将被合并至主分支并触发自动化构建部署。

## 常见问题

问：部分链接访问时返回 404 或连接超时，应该如何处理？

答：由于外部资源站点的可用性不受本项目控制，您可以通过以下方式处理：首先确认本地网络环境是否可正常访问目标域名；若确认为链接失效，请在 GitHub Issues 中标记该链接的 `id` 和访问时间，项目维护者将定期核实并移除或替换长期失效的链接。您也可以自行在本地数据文件中将对应条目的 `status` 字段标记为 `inactive` 并提交 PR。

问：批次号和文章编号的编码规则是什么？如何理解当前批次（第 13/56 批）？

答：批次号格式为 `第 X/Y 批`，其中 X 表示当前处理顺序号，Y 表示该项目计划收录的总批次数。文章编号（即 URL 末尾的数字）由源站生成，本项目仅做原始记录，不对编号含义进行额外解析。每个批次对应一个独立的 JSON 数据文件，文件名包含起始编号与结束编号范围，便于定位。

问：是否支持添加自定义分类或标签？如何操作？

答：支持。您可以在数据文件的每条记录中添加 `tags` 数组字段，并在 `src/config/categories.js` 中注册新的分类名称与显示颜色。前端界面会自动读取配置并渲染对应的筛选按钮。如需新增顶级分类，请同步更新 `docs/user-guide.md` 中的分类说明。
