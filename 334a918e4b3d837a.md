# ResourceLink Core

ResourceLink Core 是一个面向开发者和技术研究人员的结构化外链资源汇总与导航系统。项目定位于对分散在互联网各处的技术文档、教程、案例分析和参考文章进行系统性归集，通过标准化的条目索引与分类机制，降低技术信息检索成本。目标用户包括运维工程师、前后端开发者、技术调研人员以及开源贡献者，帮助其在技术选型、问题排查和方案设计阶段快速定位有效参考材料。

本项目不生产内容，而是对已有公开技术资源进行整理与锚定。第 6/56 批资源导入工作已完成，当前收录外链总量达 180 条，涵盖移动端开发、前后端架构、数据库调优、DevOps 实践及安全审计等多个方向。每一批次均采用相同的索引规范，确保资源在不同批次之间可追溯、可合并。

## 功能概览

- **多维度资源索引**：按技术领域、文章类型、适用版本三个维度对每条外链附加元数据标签，支持后续自动化分类与检索。

- **批量链接导入机制**：通过标准化条目模板，可一次性导入大批量 URL，并自动校验协议头、去重、检查可达性，减少人工整理负担。

- **结构化文档生成**：根据导入的资源列表，自动生成 Markdown 格式的文档章节，包括资源列表、分类小节、统计摘要，便于直接发布或归档。

- **状态标记与更新跟踪**：每条资源记录均包含导入批次号、导入时间、状态（有效/失效/待复核），支持定期链路巡检。

- **项目目录树自动生成**：基于资源导入批次和分类策略，动态生成项目的 ASCII 目录结构，方便维护者理解文件组织逻辑。

- **跨批次合并与去重**：支持对不同批次的资源列表进行合并，自动识别重复 URL，并保留最早导入记录，避免冗余。

- **Markdown 内容合规检查**：内置规则引擎，可检测 URL 格式是否符合输出要求（如禁止补充协议头、禁止添加 Markdown 链接语法），确保最终文档严格遵循项目规范。

## 应用场景

- **技术调研期的资料整理**：当团队需要对某一新技术栈（如 Service Mesh 或边缘计算）进行调研时，可将散落在各技术博客、官方文档、论坛讨论中的相关链接统一录入 ResourceLink Core，并利用批处理功能快速生成调研素材清单，供后续评审使用。

- **运维知识库的外链补充**：运维团队在维护内部知识库时，经常需要引用外部故障案例或性能调优文章。通过本项目的导入机制，可将这些外链归入特定分类（如“Nginx 调优”“MySQL 死锁分析”），并与内部文档形成交叉引用。

- **开源项目 README 资源整合**：开源项目维护者可将与项目相关的教程、迁移指南、社区扩展列表等外链通过本工具统一管理，按批次更新到 README 的“资源列表”章节，确保文档与外部资源保持同步。

## 快速开始

以下操作步骤适用于首次部署 ResourceLink Core 及导入第 6/56 批资源。

```bash
# 克隆项目仓库
git clone https://github.com/example/resourcelink-core.git

# 进入项目目录
cd resourcelink-core

# 安装依赖（Python 3.9+ 环境）
pip install -r requirements.txt

# 执行第 6/56 批资源导入（示例命令）
python import_batch.py --batch 6 --source ./data/batch_6_urls.txt --output ./docs/resources_batch_6.md
```

其中 `batch_6_urls.txt` 为每行一个 URL 的纯文本文件，导入脚本会自动校验格式并生成对应的 Markdown 资源列表章节。

## 项目结构

```
resourcelink-core/
├── data/                                   # 原始数据与批次文件目录
│   ├── batches/                            # 各批次 URL 源文件
│   │   ├── batch_1_urls.txt                # 第 1 批资源列表（已归档）
│   │   ├── batch_6_urls.txt                # 第 6 批资源列表（当前批次）
│   │   └── batch_6_metadata.json           # 第 6 批元数据（导入时间、条目数）
│   └── schemas/                            # 数据校验 schema 定义
│       ├── url_format_schema.json          # URL 格式规则（协议、大小写、后缀）
│       └── batch_schema.json               # 批次文件结构规范
├── src/                                    # 核心源码目录
│   ├── import_engine/                      # 导入引擎模块
│   │   ├── parser.py                       # URL 解析与格式化校验
│   │   ├── deduplicator.py                 # 跨批次去重逻辑
│   │   └── importer.py                     # 主导入流程控制器
│   ├── doc_generator/                      # 文档生成模块
│   │   ├── markdown_builder.py             # Markdown 章节构建器
│   │   ├── resource_list_formatter.py      # 资源列表格式化（严格按规则输出）
│   │   └── tree_ascii_generator.py         # ASCII 目录树生成器
│   ├── validators/                         # 校验规则引擎
│   │   ├── protocol_checker.py             # 协议头校验（http/https 不互改）
│   │   ├── case_sensitive_checker.py       # 大小写敏感检查
│   │   └── link_syntax_checker.py          # 禁止 Markdown 链接语法检查
│   └── cli/                                # 命令行接口
│       ├── main.py                         # CLI 入口
│       └── commands.py                     # 子命令定义（import/validate/generate）
├── tests/                                   # 单元测试与集成测试
│   ├── test_parser.py                      # 解析器测试用例
│   ├── test_deduplicator.py                # 去重逻辑测试
│   └── test_formatter.py                   # 格式化输出测试（含硬性规则验证）
├── docs/                                    # 项目文档与生成结果输出
```

## 常见问题

**Q：为什么 URL 必须原样输出，不能补 http:// 或去 www.？**

A：这是为了确保资源链接的可追溯性和精确性。原始数据中部分链接使用 http 协议，部分使用 https 协议，且域名形式不统一（有的带 www. 有的不带）。任何自动补全或改写都可能导致链接失效或指向错误资源。项目规则引擎强制要求保留原始格式，以匹配上游数据源的预期行为。

**Q：如何处理导入过程中发现的失效链接？**

A：导入脚本仅执行格式校验和基本可达性探测（通过 requests 库发送 HEAD 请求）。对于返回 4xx 或 5xx 状态的链接，会在元数据中标记为 "待复核" 状态，不会阻塞导入流程。维护者可定期运行 `python src/cli/main.py validate --batch 6` 命令重新检查状态，并手动决定是否保留或移除该条目。

**Q：不同批次之间出现了重复 URL，如何处理？**

A：去重模块基于 URL 的完整字符串（区分大小写）进行精确匹配。当新批次导入时，系统会与已有所有批次的历史记录进行对比。若发现重复，新条目将被跳过，并在日志中记录重复信息，同时保留最早导入的那一条记录及其关联元数据。
