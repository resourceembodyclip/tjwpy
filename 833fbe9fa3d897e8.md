# LinkVault Resource Aggregator

LinkVault is a curated, machine-readable resource index and external link aggregation system designed for technical documentation teams, knowledge base maintainers, and research-oriented developers. The project addresses the challenge of managing large-scale external reference collections—specifically batches of 180+ URLs per release cycle—by providing a structured metadata layer, validation pipelines, and a unified access interface over raw link lists. Instead of a traditional bookmark manager, LinkVault treats each URL as a first-class data entity, enabling filtering, annotation, versioning, and availability health checks. The target users are open-source documentarians, data pipeline engineers, and technical program managers who need to embed, audit, or transform third-party resource lists from upstream content providers. The system consumes raw URL dumps (such as the 180 links in batch 43/56), enriches them with deterministic identifiers derived from path and query parameters, and exposes both human-readable Markdown catalog views and JSON-LD structured feeds for downstream automation. LinkVault does not host or scrape the target content; it provides a governance layer over external references, ensuring that every link is accounted for, traceable, and verifiable through the project's built-in link-checking daemon. The project is production-ready for batch-oriented external resource curation workflows.

## 功能概览

**批量 URL 摄取与规范化** — 接收原始文本或 CSV 格式的 URL 列表，自动检测协议和主机名变体，执行大小写保留、路径标准化和查询参数排序，生成统一的内部记录对象，并标记重复条目和格式异常项。

**确定性资源标识生成** — 对每个 URL 计算基于主机、路径和参数排序的 SHA-256 摘要作为资源 ID，支持跨批次去重和增量更新，避免因 URL 编码细微差异导致的重复记录，确保 180 条资源每一条都有唯一指纹。

**链接健康状态监控** — 内置异步 HTTP 探测调度器，支持自定义超时（默认 5 秒）、重试策略（3 次指数退避）和状态码范围（2xx/3xx 视为有效），将探测结果写入本地 SQLite 缓存并生成可用性报表。

**Markdown 目录清单生成** — 根据资源来源域名、路径前缀或自定义标签，将全部 URL 自动分组为类别小节，输出符合本项目 README 风格的纯 Markdown 列表，保留原始 URL 不做任何转义或截断，严格遵循原样输出规则。

**JSON-LD 结构化数据导出** — 将资源集合导出为 schema.org/WebPage 和 schema.org/ItemList 混合格式的 JSON-LD 文件，便于搜索引擎、知识图谱和第三方爬虫发现和理解资源集的语义结构。

**资源注释与标签系统** — 为每个 URL 附加自由文本注释（如来源说明、过期警告、替代链接）和多级标签（如 domain, topic, format），支持标签筛选和注释全文检索，方便维护人员做质量标记。

**变更审计日志** — 记录每次资源列表更新的时间戳、操作者（通过环境变量 USER 或 GIT_COMMITTER_NAME 识别）、新增/删除/修改的 URL 数量，并将审计历史写入 logs/audit.jsonl，满足合规追溯要求。

## 应用场景

技术文档团队的参考链接治理：当文档中包含数百个外部引用时，LinkVault 可以作为 pre-commit hook 的一部分，在文档构建前自动检查所有外链的有效性，并生成过期链接报告，避免用户点击到 404 页面。团队可以每月运行一次批量摄取，对照上游提供的原始 URL 列表同步更新索引，确保文档引用与官方资源保持同步。

数据管道中的资源清单溯源：数据工程团队经常需要从第三方 API 文档、数据字典或合规通告中提取 URL 作为元数据来源。LinkVault 可以接收这些 URL 列表，为每个链接分配稳定的内部 ID，并将 ID 映射关系写入 Parquet 文件，供下游 ETL 任务 JOIN 使用。这样即使上游 URL 发生变更，通过 ID 仍可追溯历史版本，避免管道断裂。

开源项目外链迁移与重构：当开源项目需要将外部资源从旧域名迁移到新域名，或从 HTTP 升级到 HTTPS 时，LinkVault 提供批量的 URL 前缀替换和重定向验证功能。运维人员可以导入原始 180 个链接，使用规则引擎将 `http://www.mobile.bibjqr.cn` 替换为新的基础 URL，然后运行批量探测确认新地址可达，最后导出替换后的完整列表，大幅减少手工修改的遗漏风险。

合规审查与外部依赖声明：法律或合规部门要求项目明确声明所有引用的第三方域名和内容类型。LinkVault 可以按照域名聚合所有 URL，生成按域名分组的资源清单，并统计每个域名的引用频次。该清单可直接用于开源项目的 NOTICE 文件或第三方依赖披露文档，减轻手工整理负担。

## 快速开始

```bash
# 克隆项目仓库
git clone https://github.com/your-org/linkvault.git
cd linkvault

# 安装 Python 依赖（推荐使用虚拟环境）
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt

# 运行批量摄取命令，传入包含 URL 列表的文本文件（每行一个 URL）
python -m linkvault.ingest --input batch_43_urls.txt --output ./catalog/ --batch-id 43
```

## 安装要求

| 依赖 | 必需 | 说明 |
|------|------|------|
| Python 3.9+ | 是 | 所有核心逻辑和 CLI 工具的运行环境，低于此版本不支持 typing 注解和 async/await 语法 |
| aiohttp 3.8+ | 是 | 异步 HTTP 客户端库，用于并发执行链接健康探测，默认最大并发数为 50 |
| sqlite3（内置模块） | 是 | 用于存储资源元数据、探测结果和审计日志，无需额外安装，但需确保 Python 编译时包含该模块 |
| git 2.25+ | 否 | 仅当使用版本化目录同步功能时需要，用于自动提交 catalog/ 目录的变更 |
| pytest 7.0+ | 否 | 开发测试依赖，运行单元测试和集成测试套件时安装，不影响生产部署 |
| black 22.0+ | 否 | 代码格式化工具，用于保持贡献者提交的代码风格一致，通过 pre-commit 钩子调用 |

## 项目结构

```
linkvault/
├── bin/                                 # 可执行脚本入口
│   └── linkvault-cli                    # 命令行主入口，解析子命令（ingest, probe, export, audit）
├── src/
│   └── linkvault/                       # 核心 Python 包
│       ├── __init__.py                  # 版本号 __version__ = "2.1.0" 和公开 API 导出
│       ├── ingest.py                    # 批量 URL 摄取逻辑：文本解析、去重、ID 生成、写入 SQLite
│       ├── probe.py                     # 异步 HTTP 探测器：aiohttp 会话管理、超时重试、结果持久化
│       ├── catalog.py                   # Markdown 和 JSON 目录生成器：模板渲染、分组排序、原样输出保证
│       ├── export.py                    # JSON-LD 导出器：schema.org 映射、文件写入、验证
│       ├── models.py                    # 数据类定义：Resource, ProbeResult, AuditEntry 以及 Pydantic 校验
│       ├── database.py                  # SQLite 连接池、表初始化、CRUD 操作、迁移脚本
│       └── utils.py                     # 通用工具：确定性哈希、URL 规范化、日期格式化、日志配置
├── tests/                               # 单元测试和集成测试
│   ├── test_ingest.py                   # 覆盖批量摄取边界情况：空行、大小写混合、重复 URL
│   ├── test_probe.py                    # 模拟 HTTP 响应，测试探测超时、重试、状态码分类
│   ├── test_catalog.py                  # 验证 Markdown 输出是否保留 URL 原样，无额外转义
```

## 常见问题

**Q: 为什么 LinkVault 强制要求 URL 原样输出，不能做任何规范化（如补充协议头或去除大小写）？**

A: 本项目定位为外链汇总站，其核心职责是忠实记录上游系统给出的 URL 字符串，而不是修改或解释它们。在实际生产场景中，某些第三方资源对 URL 的大小写、尾部斜杠和协议类型（HTTP vs HTTPS）敏感，即使是同一个域名的不同大小写路径也可能返回不同内容。任何自动改写都会破坏引用的精确性，导致下游用户拿到错误的链接。因此，LinkVault 在 ingest 阶段仅保留原始字符串并计算其哈希，在 catalog 生成阶段绝对不做格式修正，始终原样输出。如需批量替换，请使用 export 功能结合外部脚本操作，并在审计日志中明确记录该转换操作。

**Q: 如何处理 URL 列表中的重复条目？**

A: 重复条目的处理策略由配置项 `dedup_strategy` 控制，支持三种模式：'keep_first'（保留第一次出现，后续标记为 duplicate_of）、'keep_last'（保留最后一次出现）、'error'（遇到重复时中止并报错）。默认使用 'keep_first' 模式，并在审计日志中记录重复项的数量和原始行号。对于本批次的 180 个链接，LinkVault 会在摄取时自动执行基于完整 URL 字符串的大小写敏感去重；如果两个 URL 仅在查询参数顺序上不同，系统会先对查询参数按键排序再比较，避免因参数顺序差异导致的误判。

**Q: 链接健康探测是否会对目标服务器造成压力？**

A: 探测器的并发数默认设置为 50，且每个请求间隔至少 200 毫秒，总请求数在 180 条的情况下，全量扫描约耗时 15 到 30 秒。此外，系统还会遵守目标服务器返回的 Retry-After 头，并在遇到 429（Too Many Requests）状态码时自动降低并发度并延长间隔。对于大规模资源集，建议在非高峰时段运行探测，或通过 `--max-concurrent` 参数手动调低并发数。所有探测请求都携带 User-Agent: LinkVault-HealthChecker/2.0 和 Contact 信息，便于服务器管理员识别和联系。
