# LinkVault 技术资源聚合系统

LinkVault 是一个面向技术研究人员、开源贡献者和内容聚合平台运营者的外链资源归集与导航系统。项目专注于将散落在各类技术社区、移动端资讯站点中的高质量文章、规范文档、工具地址进行结构化整理，并通过统一的检索与分类视图对外提供访问服务。LinkVault 本身不生产内容，而是通过严谨的链接校验、时效性监控和标签化组织，帮助用户在海量信息中快速定位所需的技术参考资料。

系统当前维护第 9 批次共计 180 个资源条目，覆盖移动开发、后端架构、数据库运维、前端工程化、信息安全等多个技术域。LinkVault 适用于个人开发者建立知识库、技术团队沉淀外部参考手册、以及社区运营者搭建内容导航页等场景。

## 功能概览

**链接健康度监控** 系统每日定时检测收录链接的可访问性，自动标记响应超时或返回 4xx/5xx 状态码的条目，并在管理后台生成异常报告。

**多维度分类索引** 每条资源支持归属多个技术标签，可按编程语言、框架版本、应用层级或问题领域进行筛选，支持组合条件查询。

**结构化元数据提取** 对每个目标 URL 执行标题解析、摘要截取和关键词频次统计，为后续检索排序提供数据支撑。

**批次导入与去重校验** 支持批量 URL 导入，自动识别重复记录并根据更新时间或命中次数保留最优版本。

**访问热度统计** 记录每个链接的点击次数与来源 IP 分布，输出周期性的热门资源榜单，辅助运营者调整推荐策略。

**快照备份与变更追溯** 对收录链接定期执行内容快照保存，记录页面结构变化，提供历史版本比对功能，便于追踪文档演进。

## 应用场景

**技术团队文档中心补充** 研发团队可在内部 Wiki 或 Confluence 之外，使用 LinkVault 搭建一个独立的外部参考链接库，将日常排查问题时频繁查阅的移动端疑难解答、SDK 变更日志、性能调优案例集中存放，减少重复搜索成本。

**开源项目 README 外链管理** 开源项目维护者可将 LinkVault 作为项目文档的延伸，把涉及依赖库、部署教程、社区讨论帖的长尾链接统一托管，避免主仓库 README 过度膨胀，同时利用 LinkVault 的检测能力确保参考链接长期有效。

**技术社区内容聚合导航** 垂直技术社区运营方可基于 LinkVault 构建外部资源聚合页，将站内精华帖与站外优质解读文章进行交叉编目，为社区用户提供一站式学习路径，提升用户停留时长。

**个人知识体系梳理工具** 技术人员可将个人浏览器收藏夹中零散的技术文章链接导入 LinkVault，通过自定义标签和备注功能构建符合自身认知结构的分类体系，并利用检索功能快速调取多年前收藏的特定解决方案。

## 快速开始

以下命令完成 LinkVault 服务的本地部署与启动，默认使用 SQLite 作为存储后端，适合测试或小规模个人使用。

```bash
git clone https://github.com/linkvault/linkvault-core.git
cd linkvault-core
pip install -r requirements.txt
python manage.py migrate
python manage.py loaddata initial_tags.json
python manage.py runserver 0.0.0.0:8000
```

访问 http://localhost:8000/admin 使用默认账号 admin / linkvault2026 登录管理后台，首次登录后请立即修改密码。生产环境部署请参考 docs/deployment.md 配置 PostgreSQL 与 Gunicorn。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Python | 3.9 - 3.11 | 核心运行时，3.12 版本存在部分异步兼容性问题，暂未完成适配 |
| Django | 4.2.8 | Web 框架，用于提供管理界面和 REST API 端点 |
| Celery | 5.3.4 | 分布式任务队列，用于执行链接健康度检测和快照更新等后台任务 |
| Redis | 7.0.x | 作为 Celery 的 Broker 和结果存储后端，同时用于缓存热点查询结果 |
| SQLite | 3.35+ | 默认数据库引擎，生产环境建议切换至 PostgreSQL 14+ 以获得更好的并发性能 |
| lxml | 4.9.3 | HTML 解析库，用于提取目标页面的标题、描述和正文结构特征 |
| requests | 2.31.0 | HTTP 客户端，配置了连接池和重试策略，用于执行链接探测 |
| django-cors-headers | 4.3.1 | 处理跨域资源共享策略，允许前端独立应用访问 API 资源 |

## 常见问题

**Q: 健康检测任务频繁触发目标站点反爬机制，如何调整检测策略？**

A: 可在 `config/settings.py` 中修改 `HEALTH_CHECK_INTERVAL` 变量值，单位为秒，默认 86400（24 小时）。同时可调整 `REQUEST_TIMEOUT` 和 `USER_AGENT_ROTATION` 参数启用随机 UA 轮换。建议将检测时段分散至凌晨低峰期，通过 `CELERY_BEAT_SCHEDULE` 中的 `cron` 表达式定制执行时间窗口。

**Q: 导入包含 1000 个以上 URL 的批量文件时出现超时，如何处理？**

A: 批量导入操作建议通过 Celery 异步任务执行，而非同步 HTTP 请求。可使用 `scripts/import_batch.py` 配合 `--async` 参数将导入任务提交至消息队列。若需同步执行，可在 Django `settings.py` 中调整 `DATA_UPLOAD_MAX_NUMBER_FIELDS` 和 `DATA_UPLOAD_MAX_MEMORY_SIZE` 参数，分别提升至 100000 和 2621440 字节。

**Q: 快照存储占用磁盘空间增长过快，有无推荐的清理策略？**

A: 系统默认保留每个链接最近 30 天的日快照，以及 30 天之前的周快照。可在 `settings.py` 中配置 `SNAPSHOT_RETENTION_DAYS` 和 `SNAPSHOT_RETENTION_WEEKS` 调整保留周期。同时可启用 `SNAPSHOT_COMPRESSION` 选项，使用 zlib 压缩存储内容，压缩率约为 60%。定期执行 `scripts/cleanup_snapshots.py` 可手动触发清理。
