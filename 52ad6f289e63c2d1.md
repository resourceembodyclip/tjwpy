# ResourceBridge 技术资源导航

ResourceBridge 是一个面向开发者和技术研究人员的结构化外链资源聚合系统。该项目对分散于多源的技术文档、教学案例、API 参考与工程实践文章进行统一采集、分类索引与快速检索，解决开发过程中因资源分散而导致的查询效率低下、优质内容被淹没的问题。ResourceBridge 定位于中大型技术团队的知识辅助工具，同时适用于个人开发者构建专属技术外链库。

## 功能概览

**批量资源导入** 支持通过数据文件或脚本批量注入外部文章链接及其元数据，自动完成去重和格式标准化。

**多维度分类标签** 为每条资源自动分配或手动指定分类标签，支持按领域、难度、资源类型等多个维度进行筛选和分组。

**全文元数据检索** 基于标题、摘要、来源域名和关键词进行快速检索，检索结果按相关性排序并高亮匹配片段。

**资源快照与状态监控** 定期检测已收录资源的可访问性状态，标记失效或重定向链接，并记录响应时间与状态码变化趋势。

**自定义收藏夹与批注** 用户可将常用资源加入个人收藏夹，并对每条资源添加私有批注或阅读笔记，支持导出为本地 JSON 或 Markdown 格式。

**外链有效性周期报告** 按周或按月生成资源健康度报告，统计有效链接、失效链接、重复条目和域名分布比例。

**导入导出互操作** 支持以 CSV、JSON、YAML 格式导入外部资源列表，同时支持将当前索引全量或增量导出，便于与其他知识管理工具集成。

## 应用场景

技术团队内部知识库建设。团队技术负责人可将成员推荐的优秀外部分享文章、官方文档和问题排查记录统一录入 ResourceBridge，形成团队共享的技术外链知识库，降低重复查找成本。

个人开发者的学习路径管理。正在学习某一技术栈（如分布式系统或前端框架）的开发者可将相关教程、案例仓库和最佳实践文章集中托管，通过检索和批注功能建立个人学习档案。

技术社区内容运营。技术社区的编辑或运营人员可使用 ResourceBridge 对投稿的外链资源进行初审、分类和可用性检测，确保对外发布的内容链接长期有效且分类准确。

开源项目文档外链维护。开源项目维护者可将项目依赖的参考文档、设计提案链接和社区讨论帖统一纳入 ResourceBridge，并在项目 README 中引用生成的资源列表，提升文档的完整性和可追溯性。

## 快速开始

以下命令用于在 Linux 或 macOS 环境下完成 ResourceBridge 的克隆、依赖安装和服务启动。

```bash
git clone https://github.com/resourcebridge/resourcebridge.git
cd resourcebridge
npm install
npm run build
npm start
```

若使用 Docker 部署，可执行以下命令：

```bash
docker build -t resourcebridge:latest .
docker run -p 3000:3000 resourcebridge:latest
```

## 安装要求

| 依赖 | 必需 | 说明 |
|------|------|------|
| Node.js 18.x 或更高版本 | 是 | 运行时环境，推荐使用 LTS 版本 |
| npm 9.x 或更高版本 | 是 | 包管理器，用于安装项目依赖 |
| SQLite 3.38 或更高版本 | 是 | 默认嵌入式数据库，无需额外安装；若使用生产环境可切换至 PostgreSQL 14+ |
| Redis 7.x | 否 | 用于缓存热点资源和会话存储，生产环境强烈建议安装 |
| Nginx 1.22 或更高版本 | 否 | 用于反向代理和静态资源缓存，生产环境推荐搭配使用 |
| Docker 24.x 及 Docker Compose 2.x | 否 | 仅在使用容器化部署时需要 |

## 贡献指南

1. 在 GitHub 上 Fork 本仓库，并于本地克隆您的 Fork 副本。请确保使用 main 分支作为基准，新建一个以 feature/ 或 fix/ 为前缀的分支进行开发。

2. 运行 npm install 安装所有开发依赖，并执行 npm run test 确认现有测试全部通过。新增功能或修复缺陷时，请同步编写对应的单元测试或集成测试用例，覆盖率不得低于原有水平。

3. 提交代码前执行 npm run lint 和 npm run format 对代码进行风格检查和自动格式化。提交信息请遵循 Conventional Commits 规范，即使用 feat:、fix:、docs:、chore: 等类型前缀。

4. 完成开发后向主仓库提交 Pull Request，并在 PR 描述中清晰说明改动目的、影响范围以及测试结果。PR 需要至少一名项目维护者进行 Code Review，通过后由维护者合并。

5. 若发现资源链接失效或分类错误，可直接在 Issue 中提交资源编号和正确信息，或按照上述流程提交修正补丁。

## 常见问题

**Q: ResourceBridge 能否自动抓取被收录页面的完整正文内容？**

A: ResourceBridge 默认仅采集目标页面的标题、元描述、关键词和来源域名，不存储完整正文，以避免版权问题和存储膨胀。若需要正文全文检索，可在配置中开启全文抓取功能，但需注意遵守目标网站的 robots.txt 规则并合理控制抓取频率。

**Q: 如何处理大量资源导入时的性能问题？**

A: 推荐使用批量导入接口，单次最多提交 500 条记录。系统内部使用批处理写入和异步索引构建。对于超过 10000 条资源的批量操作，建议通过 scripts/seed.js 脚本分批执行，并配合 Redis 缓存降低数据库查询压力。

**Q: 资源健康度检测的频率和超时时间如何调整？**

A: 健康度检测的频率和超时时间可在 config/default.yaml 中的 monitor 段落调整。checkInterval 字段控制检测周期（单位分钟），timeout 字段控制单次请求超时（单位毫秒）。生产环境建议将检测频率设置为每 720 分钟一次，超时设置为 5000 毫秒，避免对源站造成过大压力。
