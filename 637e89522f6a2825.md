# WebLink Navigator

WebLink Navigator 是一个面向技术研究者和信息分析人员的结构化外链资源聚合与导航系统。该项目并非搜索引擎，也不生产原创内容，而是通过对特定领域内的高价值外部链接进行人工筛选、分类整理和元数据标注，构建一个可快速检索、可追溯来源的参考信息枢纽。

项目定位为“技术资源的路径指引工具”，主要解决以下问题：技术从业者在阅读移动端技术文章、追踪特定平台动态或进行竞品分析时，常常面临链接散落、来源模糊、历史版本难以回溯等困境。WebLink Navigator 通过统一的条目管理和状态标记机制，帮助用户从杂乱的书签或零散的阅读记录中解脱出来，形成体系化的外链知识库。目标用户包括移动开发工程师、技术文档撰写者、产品经理以及信息安全方向的独立研究者。

## 功能概览

**资源条目化入库**：每个外部链接均以独立条目形式存储，支持自定义标签、收录时间、原始来源域名等元数据字段，便于后续过滤与检索。

**来源域名聚合视图**：系统自动解析每个 URL 的根域名并进行归类聚合，用户可按域名维度浏览所有关联条目，快速定位特定来源站点的全部资源。

**移动端适配预览**：项目内置轻量级链接预览组件，针对移动端页面特点进行渲染优化，用户在不离开导航站的情况下即可判断目标页面内容是否匹配需求。

**条目状态标记体系**：支持为每个链接设置“有效”、“失效待查”、“内容变更”、“重复收录”等状态标记，帮助维护团队持续清理和更新资源池。

**批量导入与导出**：提供基于文本列表的批量链接导入接口，同时支持将当前条目库导出为 JSON 或 CSV 格式，便于与其他知识管理工具进行数据交换。

**全文检索与过滤**：基于条目标题、标签、来源域名和收录备注构建轻量级倒排索引，支持多条件组合过滤，在数百至数千条级别下保持毫秒级响应。

**收录日志与变更追溯**：每次新增、修改或删除条目均记录操作时间与操作者标识，形成完整的资源库变更日志，方便团队协作审计。

## 应用场景

技术团队内部知识库建设：移动端开发团队在长期维护技术周报或月度技术汇总时，可将 WebLink Navigator 作为底层链接管理工具。团队成员各自发现的优质外部文章统一入库，再由编辑角色进行标签补充和分类整理，最终生成结构化的团队技术周报素材池。

竞品动态跟踪与信息聚合：产品经理或市场分析人员可针对特定竞争对手的移动端页面（如 H5 活动页、产品更新说明页）进行链接收录，通过定期检查条目状态标记的变化，快速感知竞品在功能描述、活动规则等方面的调整节奏。

技术文档写作的参考来源管理：技术博客作者或开源项目文档维护者在撰写技术方案对比、框架选型分析等内容时，需要引用大量外部参考链接。WebLink Navigator 可作为参考来源的临时缓存与标注工具，在成文后统一导出引用列表，避免手动整理链接时出现的遗漏或格式错误。

个人阅读流的信息沉淀：技术爱好者日常通过移动端阅读大量 H5 技术文章，但浏览器的收藏夹管理能力有限。使用该项目可将有价值的单篇文章统一收录，配合标签体系（如“性能优化”、“安全加固”、“CSS 新特性”）构建个人化的阅读档案，方便后续按主题回溯。

## 快速开始

以下步骤适用于首次部署 WebLink Navigator 的开发者。项目基于 Node.js 运行时构建，前端采用静态生成方案，所有数据存储于本地 JSON 文件，无需配置外部数据库。

```bash
# 克隆项目仓库至本地
git clone https://github.com/weblink-navigator/weblink-navigator.git

# 进入项目工作目录
cd weblink-navigator

# 安装项目依赖（使用 npm）
npm install

# 以开发模式启动本地服务，默认监听端口 3000
npm run dev
```

启动成功后，访问控制台输出中显示的本地地址（通常为 http://localhost:3000 ）即可进入导航站主界面。首次启动时系统会自动生成示例条目数据，用户可通过界面上的“导入”按钮批量替换为自己的链接列表。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Node.js | 16.20.0 或更高 | 项目运行时与构建工具链的基础环境，推荐使用 LTS 版本 |
| npm | 8.0.0 或更高 | 依赖包管理工具，随 Node.js 一同安装 |
| 现代浏览器 | 最近两个主要版本 | 前端界面基于 Chromium 内核特性开发，支持 Chrome、Edge、Firefox 最新版 |
| 文件系统读写权限 | 必需 | 项目所有数据以 JSON 格式存储于本地文件系统，需确保运行用户拥有读写权限 |
| 网络连通性 | 用于外链预览 | 预览功能需访问外部链接，需保证运行环境可访问公网；离线模式下预览功能不可用 |

## 项目结构

```
weblink-navigator/
├── data/                           # 数据存储目录（所有条目与配置均落地于此）
│   ├── entries.json               # 主条目库，包含所有链接及元数据
│   ├── tags.json                  # 标签字典，记录标签名称、颜色及使用计数
│   ├── audit.log                  # 操作审计日志，追加写入，按行记录时间戳与动作
│   └── schema/                    # 数据结构的 JSON Schema 定义，用于校验
│       └── entry.schema.json      # 单条目的字段类型、必填与约束规则
├── src/                            # 源代码目录
│   ├── core/                      # 核心业务逻辑模块
│   │   ├── entry-manager.js       # 条目的增删改查、状态变更与校验入口
│   │   ├── tag-engine.js          # 标签的合并、拆分与重命名逻辑
│   │   └── import-export.js       # 批量导入（TSV/JSON）与导出（JSON/CSV）实现
│   ├── web/                       # 前端界面源码（基于原生 JavaScript + CSS）
```

## 贡献指南

1. 查阅现有议题与项目看板：在提交新的功能请求或问题报告之前，请先浏览 GitHub Issues 与 Projects 面板，确认是否有已有的讨论或正在进行中的工作，避免重复劳动。

2. 分支开发与提交规范：所有代码修改应在个人分支上进行，分支命名采用 `feature/` 或 `fix/` 前缀。提交信息遵循 Conventional Commits 格式（如 `feat: add batch import progress bar`），便于自动生成变更日志。

3. 补充或更新测试用例：任何对核心逻辑（条目管理、标签引擎、导入导出）的修改均需同步更新对应的单元测试。新增功能必须附带至少一个正向测试和一个边界条件测试。

4. 文档同步更新：如果修改涉及用户界面交互、API 接口或配置文件格式变更，必须同步更新 docs 目录下的对应文档。文档更新应在同一 PR 中提交，与代码变更一同审查。

5. 提交 Pull Request 并通过检查：完成开发后，向主仓库的 `main` 分支提交 PR。PR 描述中需明确说明变更目的、影响范围以及手动测试的步骤。所有 CI 检查（包括测试通过、代码格式检查、文档构建）均需为绿色状态方可合并。

## 常见问题

问：项目是否支持在线多人协作编辑条目库？

答：当前版本的设计定位为单机工具或团队内通过版本控制系统（如 Git）共享数据文件。数据全部存储于本地 JSON 文件，不支持内置的实时多用户协同编辑。若团队需要多人同时维护，建议将 data/ 目录纳入 Git 仓库，通过分支与合并流程进行协作，或自行开发基于该数据格式的服务端同步层。

问：导入大量链接时页面出现卡顿，如何优化？

答：导入操作涉及文件读取、数据校验和索引重建，对于超过 500 条记录的批量导入，建议使用命令行工具而非界面导入。项目提供 `npm run import -- --file=path/to/links.txt` 命令，该命令绕过前端渲染层，直接操作数据文件，可处理数千条记录。导入完成后重启界面服务即可。

问：预览外链时提示“无法访问”，但链接在浏览器中直接打开正常，是什么原因？

答：预览功能通过后端服务发送 HTTP 请求获取目标页面摘要信息。部分站点会校验请求头中的 User-Agent 或来源域名，导致服务端请求被拒绝。解决方法：在 `src/server/dev-server.js` 中配置预览请求的代理头信息，或切换到“仅显示链接”模式，完全依赖用户手动点击访问。该限制不影响条目管理功能。
